## 3. 这个项目到目前为止的大致进度

从仓库历史和实验产物看，到 `2026-03-28` 为止，项目主线可以概括成四段。

### 3.1 起步阶段：先把 `监督学习阶段` 基础设施搭起来

大约对应 `2026-03-10` 到 `2026-03-13`。

这几天主要工作不是“追求模型上限”，而是让实验能跑、能比较、能复盘：

- 建训练和验证管线
- 调整 loader、线程、内存与日志行为
- 清理旧实验残留
- 加入 `监督学习阶段` A/B runner
- 把 formal 与 A/B 入口拆开
- 处理 Windows 下 collate / pickle / worker 的工程问题

这一阶段的核心成果不是哪条模型曲线，而是：

- 你有了可以连续开展研究的实验底座

### 3.2 中段：开始怀疑“超参 winner”本身是否可信

大约对应 `2026-03-12` 到 `2026-03-23`。

这段时间，项目从“调几个超参”逐渐变成“修实验制度”：

- 从单因素 A/B 推进到 `AB23`、`AB234`、再到 `AB1234`
- 开始关心因子之间是不是强耦合，而不是默认它们独立
- 开始修 A/B fairness
- 引入更 action-aware 的评分
- 重视 selector 本身的语义，而不是只盯着原始 loss

这一阶段说明，你的关注点已经不是：

- “哪个选项在当前表里赢了”

而是：

- “这张表是不是公平”
- “winner 是否由错误口径制造出来”
- “实验因子是不是被拆错了”

### 3.3 后段：`监督学习阶段` 从调参进入“选模制度化”

大约对应 `2026-03-21` 到 `2026-03-27`。

这段时间，最重要的推进不是主干网络改了，而是整个 `监督学习阶段` 的证据链越来越严：

- `P0 fidelity` 跑起来了
- leakage 问题被确认，并触发 clean rerun
- `P0 top3` 被重新冻结
- `P1` 口径从旧的总 loss 判断转向 `policy_quality`
- `A2y` 上 `rank / opp / danger` 的内部 shape 被单独 micro-AB 冻结
- `P1` 主流程进入 `solo -> pairwise`
- 同时并行做 loader/affinity 吞吐对照

这一步很关键，因为项目在这里开始明确：

- 不是“多跑一点超参就会更强”
- 而是“先保证比较口径正确，再决定谁值得留下”

### 3.4 当前状态：还在 `监督学习阶段` 收尾，但已经开始补下游接口

到 `2026-03-28` 为止：

## 4. 这个项目里，问题通常是怎么被发现的

从现有文档看，你的典型工作方式不是“先写代码，再被报错教育”，而是下面这种链路：

1. 先对实验结果或口径产生怀疑  
2. 再回头质疑问题定义是否成立  
3. 然后要求模型把这个怀疑展开成可验证实验  
4. 再根据结果修正实现、文档和默认路线

也就是说，真正的起点通常不是：

- 代码 bug

而是：

- 定义不对
- 比较不公平
- 指标与最终目标不一致
- 历史结论可能被 leakage、旧缓存或错误拆因子污染

这是一个很重要的能力信号。因为很多人做研究型工程时，卡住的地方不是不会写实现，而是不知道应该怀疑什么。

## 5. 你在这个项目里实际发挥的优势

虽然你没有手写代码，但从仓库痕迹看，你承担的并不是“提需求的人”，而是更像研究方向和判断质量的把关者。

### 5.1 你在定义问题

你真正做的不是让模型代你写脚本，而是不断决定：

- 什么问题值得继续追
- 什么结果不能直接信
- 哪个对比不公平
- 哪种口径才更接近最终目标

这类工作本质上不是 coding，而是 research framing。

### 5.2 你在发现结构性问题

你比较强的地方，不是在已有框架里做局部最优，而是能发现：

- 因子拆分错了
- 评分语义重复计分
- calibration 结果被误读
- 旧的 `P1` winner 其实不该继续当主线

这类问题如果没人先指出来，模型往往会沿着现有实现继续优化，而不会主动把它上升成“主问题”。

### 5.3 你在判断什么证据可信

你不是只看一轮 winner，而是会继续追问：

- 这个结论是不是短预算假象
- 是不是比较口径不一致
- 是不是旧日志和新 selector 混用了
- 是不是 auxiliary tax 误导了判断

这意味着你在做的是证据筛选，而不是结果搬运。

## 6. `GPT-5.4` / `Codex` 在这个项目里真正擅长什么

从现有工作痕迹看，模型侧的强项很稳定。

### 6.1 把一个判断快速展开成完整执行链

一旦方向被你确定，模型特别适合快速补全：

- 脚本
- runner
- 配置
- 测试
- 文档
- 日志解析
- 汇总报告

这让“一个合理想法”更容易在同一天变成“一个真正跑起来的实验”。

### 6.2 大规模检查一致性

模型很适合做这种事情：

- 对比多份文档是否口径一致
- 审查代码和 selector 实现是否一致
- 批量追踪日志与产物
- 把一个口径修正扩散到测试和说明文档

这种工作人也能做，但又慢又容易漏。

### 6.3 降低实验工程摩擦

在这个项目里，模型最大的工程价值不是“替你思考”，而是：

- 让你能把更多精力放在判断题，而不是手工实现题

## 7. 人和模型各自的优势，放在一起后新增了什么

最有价值的地方，不是“人负责高层，AI 负责低层”这么粗糙，而是：

- 人负责发现真正值得问的问题
- 模型负责把问题迅速变成可执行、可复现、可比对的证据链

### 7.1 人的优势

- 发现定义层面的不对劲
- 识别哪些结论值得怀疑
- 决定最终目标到底是什么
- 判断是否该继续投入某条路线

### 7.2 模型的优势

- 快速展开实现
- 快速做对照和补丁
- 快速让脚本、测试、文档同步
- 快速压缩工程摩擦

### 7.3 协同后的真正增益

协同后的优势不是简单相加，而是让你能进入一种新的工作模式：

- 先用人的判断发现“主问题”
- 再用模型迅速验证多个备选方向
- 再由人筛掉伪结论
- 最后由模型把最终方向完整落地

这会显著提升两件事：

- 问题迭代速度
- 从想法到证据的转化速度

## 8. 从这个项目里总结出来的模型使用经验

### 8.1 不要把模型当最终判断者，要把它当加速器

模型很适合：

- 展开
- 检查
- 实现
- 归纳

但不应默认让它替代：

- 研究目标定义
- 结论可信性判断
- 资源分配优先级

### 8.2 先定义比较口径，再让模型写代码

你当前项目里很多关键推进，都来自先把下面的问题问清：

- winner 按什么判
- 哪个指标只是诊断
- 哪个比较不公平
- 哪个实验因子不能拆开看

如果这一步没做，模型写得越快，可能只是更快把错误口径制度化。

### 8.3 让模型一次性修改整条证据链

比较有效的用法不是：

- 让模型只改一个函数

而是：

- 同时改代码、测试、文档、分析脚本和运行默认值

这样更容易避免“实现变了，但说明和验证口径没同步”的问题。

### 8.4 保留原始实验产物，不要只保留结论

当项目进入研究阶段后，最重要的不是一句话 winner，而是：

- 原始 run 目录
- summary
- state
- selector 版本
- 文档化解释

模型可以帮你整理，但你必须要求它保留证据，不然以后很难复盘。

### 8.5 要主动防止模型顺着旧实现滑下去

模型默认倾向于：

- 延续现有结构
- 修局部 bug
- 在现有语义下做更整齐的实现

所以如果问题本身在“定义层”而不在“实现层”，你需要主动指出：

- 这里要怀疑的是问题定义，不是函数细节

### 8.6 把模型当作研究搭档，而不是自动驾驶

最好的使用方式不是：

- 给一条命令，等最终答案

而是：

- 让模型做快速展开
- 人持续插入判断、修正和重定向

也就是：

- 模型负责高频执行
- 人负责低频但高价值的方向修正

### 8.7 要定期重审阶段图，删除已经失去下游读者的中间层

`监督学习阶段` 里的 `P2` 是一个很典型的反例。

它刚引入时并不是完全没道理。旧设计里，`P2` 负责两件事：

- 在 `P0/P1` 产出的稳定训练曲线里离线比较 `K1_best_loss / K2_best_acc / K3_best_rank / K4_latest`
- 把不同 selector 与 checkpoint 口径去重，保留 `2 ~ 4` 个候选，送去当时规划里的 `P3(监督学习阶段 transfer)`

在那个阶段，这样做的出发点是合理的：

- `监督学习阶段 transfer` 比较贵
- 先用离线口径缩小候选，看起来像是在节省下游预算
- 当时还没有当前这套已经收口的 `winner_refine -> formal` 主线

但后面的主线已经变了：

- `P1` 收口成 `calibration -> protocol_decide -> winner_refine`
- `formal` 会对最终 `P1 winner` 再跑一轮正式 checkpoint compare
- `formal` 还会直接发布 canonical checkpoint，供下游 `监督学习阶段` 使用

这时再保留 `P2`，问题就出现了：

- 它不再改变任何主线决策
- `formal` 不读取它的输出
- 它只留下一个看起来完整、实际上没有下游消费者的中间层

这类错误很容易由人机协同一起制造出来。模型会天然沿着旧结构继续补全：

- 旧阶段名还在，它就会继续维护状态字段、文档和摘要
- 旧选择器还说得通，它就会把这层解释得越来越整齐
- 如果人只检查“局部是否合理”，而不检查“删掉这一层后到底有什么会坏”，这个结构就会长期存活

`P2` 从“有点道理”走到“完全错误”，并不是因为实现坏了，而是因为它原本服务的下游关系已经消失了，但代码和文档里的阶段名还活着。

这个案例对应一条很具体的方法论：

- 每次主线阶段图变化后，都要重新问三件事：
- 这个阶段读什么输入
- 这个阶段写什么输出
- 到底是谁在读它的输出

如果第三个问题答不出来，就不该继续优化它的实现，而应该直接删除它。

这也是人和模型分工里很关键的一点：

- 模型擅长沿着现有结构做局部一致化
- 人必须负责定期做“阶段图审计”，识别哪些层已经变成没有消费者的僵尸结构

否则模型的高执行效率，反而会更快把一个过期设计包装成看起来很成熟的系统。

## 9. 如果未来考虑写论文，这套材料有什么价值

如果后续模型效果真的足够好，这些文档并不只是“个人感想”，而是论文准备的前置资产。

它们至少有三种价值：

- 帮你保留真实研究路径，而不是事后美化
- 帮你区分哪些设计是拍脑袋，哪些设计是证据驱动
- 帮你在写论文 related work / ablation / methodology 时，回到真实问题链条

尤其是下面三类材料，以后会很值钱：

- 哪些问题是怎么被发现的
- 哪些对比为什么不公平
- 哪些设计最后为什么被冻结为默认值
