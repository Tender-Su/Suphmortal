# 2026-04-06：监督学习 Oracle 退役复盘

## 1. 这轮讨论最后收口成了什么

这轮最重要的结论只有一句：

- 监督学习阶段不再继续 Oracle 路线

项目当前只保留两大阶段：

- 监督学习阶段
- 强化学习阶段

监督学习阶段已经完成，当前写死的结论是：

- 当前正式 winner：`anchor*1.0`
- 当前第一替补：`opp_lean*0.85`
- 当前 canonical checkpoint 仍沿用历史文件名 `sl_canonical.pth`

Oracle 如果以后还要做，只会去强化学习阶段里重新设计，而不是再回监督学习阶段里补一个旧式续跑。

## 2. 为什么最后放弃了监督学习 Oracle

前面的讨论其实经历了一个很典型的收缩过程。

一开始的思路是：

- 既然 `Suphx` 用过 Oracle guiding
- 那我们也许应该在监督学习阶段里做 `oracle-dropout`
- 再比较比例和曲线

这条路一开始看上去很合理，因为它像是在“给模型一个更强老师”。但继续往下想，问题就出来了。

真正的冲突点是：

- 监督学习的动作标签来自人类牌谱
- 人类做决策时看不到全局隐藏信息
- 如果模型训练时却能看到 Oracle 信息，那么模型在很多局面下会天然比标签更懂局面

这会带来一个结构性问题：

- 模型不是在学“人类在这个观测下会怎么做”
- 而是在学“一个看见全局信息的强观察者，怎样去拟合一个看不见全局信息的人类动作”

这两者并不一致。  
一旦 Oracle 信息足够强，监督学习就很容易变成一种带偏差的倒退学习：  
模型已经知道更多，却被迫去拟合一个信息更少的标签。

所以后面的判断逐渐收紧为：

1. 监督学习里继续加 Oracle，不是自然增强，而是在引入标签语义冲突。
2. 如果强行做，最稳的形态也只能是“临时扶手”，而不是主线。
3. 既然它不能成为主线，就不值得继续消耗训练预算和工程复杂度。
4. Oracle 真正更适合出现的位置，是强化学习。

原因也很直接：

- 强化学习优化的是回报，不是人类动作标签
- 这时全局信息可以服务于价值估计、训练稳定性或 teacher/critic 设计
- 不会再和“人类看不到这些信息”发生正面语义冲突

所以最终不是“Oracle 完全没价值”，而是：

- 它不该待在监督学习阶段
- 它应该挪去强化学习阶段

## 3. 这次讨论里人类擅长什么

这轮最关键的人类贡献，不是在代码层面，而是在问题定义层面。

人类最先抓住的是那个最本质的矛盾：

- 训练时看得见全局
- 标签却来自看不见全局的人

这不是一个局部超参问题，也不是“多跑几个 A/B 就知道”的问题，而是一个训练目标是否自洽的问题。

人类在这类问题上明显更擅长：

- 发现目标函数和数据来源之间的语义冲突
- 区分“听起来更强”和“逻辑上更对”
- 在证据还不完备时，先抓住真正该追问的核心矛盾
- 对路线做减法，阻止项目沿着一个看似高级但本质不稳的方向继续扩张

如果没有这部分判断，agent 很容易继续沿着既有思路把 `oracle-dropout` 做得越来越完整，最后得到一条工程上顺滑、但研究问题本身已经歪掉的路线。

## 4. 这次讨论里 agent 擅长什么

agent 的强项主要体现在另外几个层面：

- 快速把仓库现状、文档口径、代码入口和论文描述串起来
- 把模糊的担心展开成可检查的结构
- 区分哪些是论文里真的说过的，哪些只是印象或混淆
- 及时指出当前文档和代码主线已经与最新判断冲突
- 在路线确定后，系统性清理旧入口、旧配置和旧说明

尤其是在这次讨论里，agent 的价值不在于替人做最终研究判断，而在于：

- 把已有仓库状态完整摊开
- 把 Suphx 的相关做法查清楚
- 把“如果继续做监督学习 Oracle，工程上会怎样展开”这件事讲清楚
- 在人类拍板后，快速把项目口径改干净

这类工作非常适合 agent：

- 资料查找快
- 代码和文档联动快
- 大量细枝末节的统一收口快

## 5. 这次协作最值得保留的方法

这轮讨论里最有效的方法不是“谁更聪明”，而是分工对不对。

较好的分工方式是：

- 人类负责抓研究问题是否成立，抓目标是否自洽
- agent 负责把现状、实现、证据和影响范围迅速摊平
- 人类做最终路线裁决
- agent 负责把裁决落到代码和文档里

这说明一个很重要的协作原则：

- agent 很适合加速收集、整理、实现和清理
- 人类仍然最适合决定“这个问题到底该不该问”

当路线判断本身出了偏差时，不能指望单纯靠更多自动实验把它纠正回来。先把问题问对，比把旧问题跑得更彻底更重要。
