# Stage 0 GRP 实验记录

本文档只讨论 `Stage 0` 的 `GRP`。重点不是罗列所有试验，而是说明：为什么要做这些对比、哪些结论已经比较稳、哪些结论还没有真正关掉。

## 当前状态

当前默认主线仍是 `384x3 fp32`。这是现阶段证据最充分、综合效果最稳的一组配置，因此后续主训练默认仍从这里出发。

但这不代表 `Stage 0` 已经彻底结束。最近补测的推理延迟表明，`512x4` 在这台机器上的单局推理成本并不高，至少不能再因为“推理太慢”而直接排除。因此，`512x4` 现在应被视为 `GRP` 的高优先级继续探索候选，而不是已经被判死刑的边界实验。

## 为什么还要继续探索

`GRP` 的作用不是单独追一个更低的验证损失，而是为后续阶段提供更可靠的全局收益建模。如果模型太小，顺位结构学不稳；如果模型太大，训练和部署成本会上升，但未必真的换来更强的下游效果。前面的实验已经证明小模型不够，也证明 `384x3` 很稳，但还没有充分证明“更大的候选一定不值得继续追”。尤其是在推理延迟已经基本可接受的前提下，接下来更关键的问题变成了：更大的模型是否能在更长训练预算和下游验证里给出稳定优势。

## 已比较的候选

当前已经实际比较过四组设置。

- 第一组是小到中等规模架构：`64x2`、`128x2`、`128x3`、`192x2`、`192x3`、`256x2`、`256x3`、`384x2`、`384x3`
- 第二组是更激进的放大：`256x4`、`384x4`、`512x3`、`512x4`、`1024x4`
- 第三组是精度：`fp64` 与 `fp32`
- 第四组是围绕 `384x3` 的训练配置：`bs=512, lr=1e-5`、`bs=1024, lr=1e-5`、`bs=1024, lr=1.5e-5`、`bs=1024, lr=2e-5`

这里的 `384x3` 表示隐藏维度 `384`、GRU 层数 `3`。

## 推理延迟

这次重新补测了真实的 `GRP` 推理延迟。测试口径是：从 `D:/mahjong_data/grp_pt/val` 抽取 `80` 局真实样本，平均前缀长度约 `10.85`，中位数 `11`，最大值 `19`；统计的是单局 end-to-end 推理时间，包含前缀构造、张量搬运、`GRP.forward` 和 `calc_matrix`。

- `192x3`：均值 `4.0461ms`，中位数 `3.6298ms`，`p90 = 5.9500ms`
- `256x2`：均值 `3.8484ms`，中位数 `3.4815ms`，`p90 = 4.9897ms`
- `256x3`：均值 `4.1802ms`，中位数 `3.7548ms`，`p90 = 5.9563ms`
- `384x3`：均值 `4.2036ms`，中位数 `3.6925ms`，`p90 = 6.0139ms`
- `384x4`：均值 `4.1101ms`，中位数 `3.7890ms`，`p90 = 5.2286ms`
- `512x3`：均值 `3.9414ms`，中位数 `3.6201ms`，`p90 = 5.2466ms`
- `512x4`：均值 `4.0096ms`，中位数 `3.8147ms`，`p90 = 4.7653ms`
- `1024x4`：均值 `4.3789ms`，中位数 `3.9409ms`，`p90 = 6.2433ms`

这组结果最重要的意义是：`256x3`、`384x3`、`384x4`、`512x3`、`512x4` 的推理时延都在大约 `4ms` 左右，差距很小。换句话说，`512x4` 目前并没有因为推理成本而被淘汰。

## 当前判断

目前比较稳的结论有四条。

第一，小模型不够。`64x2`、`128x2` 这一档的容量明显不足，不能作为主线。

第二，`256x3` 和 `384x3` 是已经被证明有效的核心候选。其中 `256x3` 更适合当效率优先备选，`384x3` 目前仍然是证据最充分的默认主线。

第三，更大的模型并不是完全没有价值。`384x4`、`512x3`、`512x4` 在离线结果上并没有形成稳定碾压，但差距也没有大到可以直接排除；现在又知道它们的推理时延仍然在可接受范围内，因此它们更像“证据还不够”的候选，而不是“已经证明不值得”的候选。

第四，`1024x4` 仍然不适合作为当前主线。它的离线结果没有形成明显优势，推理时延也开始抬升，因此继续投入它的优先级不高。

## 下一步探索

`Stage 0` 现在最值得继续做的，不是再盲目扩搜索空间，而是把最有希望的候选做长预算复验。当前优先级建议如下：

- 第一优先级：`384x3` vs `512x4`
- 第二优先级：`384x3` vs `384x4`
- 第三优先级：`384x3` vs `512x3`

这些对照需要满足三个条件：

- 使用相同数据范围和相同训练预算
- 使用相同选模口径
- 最后不仅看 `GRP` 自身验证结果，还要看对下游监督学习 / 强化学习的实际影响

在这些更强证据出现之前，默认主线仍然保持 `384x3 fp32`，但文档和计划都不应把 `Stage 0` 写成已经完全封闭的问题。

## Checkpoint 口径

当前 `GRP` 至少保留三个角色不同的 checkpoint：

- `best_loss`：默认下游使用
- `best_acc`：作为受控对照候选
- `latest`：只用于续训

这样做的原因很简单：`best_loss` 更适合作为主线默认，而 `best_acc` 可能在某些下游实验里值得验证，但不应在没有证据时静默替换主线。
